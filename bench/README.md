# bench/

Reproducible time-to-converge benchmark for the launch claim that scampi
is fast. Tracks issue #260.

**Scenario:** "deploy nginx + 5 templates + restart service to 3 hosts"
— recognisable to an Ansible-fluent reader on a drive-by, not a
contrived microbench.

**Approach:** both tools run against the same 3 fresh LXCs on the same
PVE host, same network, same kernel, same workload. The delta tells
the story; absolute numbers will vary by hardware/network.

## Layout

bench/
- README.md                       — you are here
- .env.example                    — env-var template (committed)
- .env                            — your values (gitignored)
- provision/
  - lxcs.scampi                   — phase 1: create the LXCs
  - bootstrap.scampi              — phase 2: perf-user + sudo + ssh key
  - snapshot.scampi               — phase 3: pristine snapshot for cold-run rollback
- teardown.scampi                 — destroy the LXCs (state = absent)
- scampi/
  - deploy.scampi                 — workload: nginx + 5 templates + service
  - templates/                    — Go templates (.tmpl)
- ansible/
  - site.yml                      — matched ansible playbook
  - ansible.cfg                   — ControlMaster, pipelining, forks
  - requirements.txt              — pinned ansible-core for the venv
  - inventory.ini                 — generated by provision.sh from .env (gitignored)
  - templates/                    — Jinja templates (.j2)
- lib.sh                          — shared env load + require_* helpers
- provision.sh                    — one-shot setup (keypair, venv, inventory, 3 phases)
- bench.sh                        — single-scenario hyperfine harness (cold + warm)
- matrix.sh                       — sweep (HOSTS, WAN_DELAY_MS) cells
- _cold_prep.sh                   — uncounted prep per cold run (rollback + start + wait + WAN)
- _apply_wan.sh                   — apply / clear netem qdisc per LXC
- results/                        — per-cell timing JSON + markdown summaries

## How it's wired

All environment-specific values (PVE host, IP range, VMID range,
node name, gateway, etc.) live in `bench/.env`, which is gitignored
so your real network details never land in the repo. Defaults and
documentation are in `bench/.env.example`, which is committed.

Two entry points, separated by responsibility:

- **`bench/provision.sh`** — one-shot setup. Sources the env, generates
  the bench SSH keypair, builds the ansible venv from pinned
  `requirements.txt`, writes `bench/ansible/inventory.ini`, and
  applies the three provision phases (`lxcs` → `bootstrap` →
  `snapshot`) in order. Idempotent — re-run after changes to
  `provision/`, `.env`, or to expand the fleet to a larger
  `BENCH_HOSTS`.
- **`bench/bench.sh`** — single-scenario timing run. Reads
  `BENCH_HOSTS` and `WAN_DELAY_MS` from the env, applies (or
  clears) the WAN netem on each LXC, then runs hyperfine cold +
  warm against scampi and ansible. Writes per-run JSON + markdown
  to `bench/results/<ts>.h<HOSTS>.{lan|wan<MS>}.*`.
- **`bench/matrix.sh`** — sweeps the cartesian product of
  `HOSTS_MATRIX` × `WAN_MATRIX` (defaults: (1, 3, 10) × (0, 100))
  by invoking `bench.sh` for each cell. The metadata file in each
  cell records its `hosts` and `wan_delay_ms` so cells can be
  compared after the fact.

The phases are split because `std.deploy` blocks within one file run
in parallel; phases 2 and 3 need the LXCs from phase 1 to exist
already, so they're separate files applied sequentially.

The provision phases, `teardown.scampi`, and `bench/scampi/deploy.scampi`
read every value via `std.env()` (with `std.parse_int()` for the
numeric ones) — there's nothing to edit in those files. To run them
by hand outside the wrapper scripts, source the env first:

```bash
set -a; source bench/.env.example; source bench/.env 2>/dev/null; set +a
scampi apply bench/provision/lxcs.scampi
scampi apply bench/provision/bootstrap.scampi
scampi apply bench/provision/snapshot.scampi
# or: scampi apply bench/teardown.scampi
```

## Pinning policy

The bench is reproducible only if every moving part is pinned.
Everything below is locked to a specific version:

- **Debian template**: `debian-12-standard_12.12-1_amd64.tar.zst`
  (in `provision/lxcs.scampi`).
- **nginx package**: `nginx=1.22.1-9+deb12u4` (single source of
  truth: `NGINX_VERSION` in `bench/.env.example`; both
  `bench/scampi/deploy.scampi` and `bench/ansible/site.yml` read it
  via env). If Debian's stable apt repo no longer carries this exact
  version (security updates ship periodically), bump the env default
  and re-baseline the numbers.
- **Ansible**: `ansible-core==2.18.5` in
  `bench/ansible/requirements.txt`, installed into a venv that
  `provision.sh` creates on first invocation. No community collections —
  `ansible.builtin` is enough.
- **scampi**: see "Which scampi binary" below.
- **hyperfine**: prerequisite on the controller, version captured in
  the run's `<timestamp>.metadata.txt`.
- **Container resources**: 2 vCPU, 2 GB RAM, 8 GB disk per LXC
  (in `provision/lxcs.scampi`).

## Methodology

Honest measurement requires fairness and disclosure. Both tools get:

- The **same** 3 target hosts (created by the `provision/` phases)
- The **same** workload intent (idiomatic in each tool, no handicapping)
- The **same** controller machine and network path
- **N runs** each (default 10), median + p95 reported, cold + warm
- **Caveats** disclosed up-front: hardware, network locality,
  controller, kernel versions

If the baseline wins on any metric, that gets reported too. The bench
page on `scampi.dev/bench` reads like a benchmark report from someone
who cares about being right, not a sales pitch.

### Cold vs warm

- **Cold**: target rolled back to the `pristine` snapshot before
  every timed run (uncounted prep). Each cold timing measures a true
  from-scratch deploy. Snapshot rollback restores byte-identical
  state — strictly better than `apt purge && reinstall` which leaves
  apt's cached metadata warm.
- **Warm**: one untimed warmup, then N timed re-runs against an
  already-converged target. Each warm timing measures the
  idempotent / no-op path — "is anything drifting?".

### SSH key

The bench uses a dedicated ephemeral keypair, **not** your personal
SSH key. `provision.sh` generates `bench/.ssh/scampi_bench{,.pub}` on first
run (gitignored), passes the public half into `provision/lxcs.scampi`
via `$SCAMPI_BENCH_SSH_PUB`, and the private half is what scampi and
ansible use to reach `perf-user@bench-*`. The keypair lives only as
long as the bench LXCs do — `scampi apply bench/teardown.scampi`
removes the LXCs and the key files become useless. `rm -rf
bench/.ssh/` to clean up the key files themselves.

## Which scampi binary

Two phases of the bench:

1. **Pre-release** (now): `./build/bin/scampi` built from the current
   main branch via `just build`. Re-run after meaningful changes to
   `engine/`, `target/ssh/`, `target/posix/`, the `lang/` pipeline,
   etc. `bench.sh` records the binary's reported version (including
   commit SHA via `scampi version`) into `<timestamp>.metadata.txt`.
2. **Post alpha.8 release**: install via `curl get.scampi.dev | sh`
   to confirm the binary published to users matches what was
   benchmarked. Set `SCAMPI=~/.local/bin/scampi` when calling
   `bench.sh` (or `matrix.sh`).

Both rounds use the same workload, same harness, same hosts, same
ansible. Numbers from round 2 are the ones that ship on the bench
page.

## How to run

### Prerequisites on the controller

- Python 3.11+
- `hyperfine` (1.x). On macOS: `brew install hyperfine`. On Debian:
  `sudo apt install hyperfine`.
- `ssh` + `ssh-keygen` (already there if you've ever used SSH)
- Ability to `just build` the scampi repo

### One-time setup

```bash
cp bench/.env.example bench/.env
$EDITOR bench/.env
```

Set at minimum:

- `PVE_HOST` — hostname/IP of your Proxmox host
- `PVE_NODE` — cluster node name (as shown in the PVE web UI)
- `PVE_GATEWAY` — gateway inside the bench LXCs
- `BENCH_IP_PREFIX` — first three octets, e.g. `192.168.1`

Sensible defaults exist for the rest (VMID base 950, IP base 200,
`perf-user`, snapshot name `pristine`, 3 hosts, LAN, 10 runs cold +
warm).

### Run

```bash
just build              # one-off; re-run after scampi changes
bash bench/provision.sh # one-off per host count; idempotent
bash bench/bench.sh     # one scenario (HOSTS + WAN_DELAY_MS from env)
```

Single-cell tunables (env vars; can also be set in `bench/.env`):

- `RUNS=5` — fewer runs for a faster smoke
- `PHASES=warm` — warm only (skip cold)
- `BENCH_HOSTS=1` — host count for this run
- `WAN_DELAY_MS=100` — egress netem delay per LXC (0 = LAN)
- `SCAMPI=/path/to/scampi`, `ANSIBLE_VENV=/path/to/venv`
- `SSH_KEY_FILE=/path/to/key` (defaults to `bench/.ssh/scampi_bench`)

### Scaling matrix

For the (1, 3, 10 hosts) × (LAN, WAN) sweep on the published bench
page, provision once at the maximum host count, then run `matrix.sh`:

```bash
BENCH_HOSTS=10 bash bench/provision.sh    # creates 10 LXCs (one-time)
bash bench/matrix.sh                      # 6 cells × cold+warm × 2 tools
```

Each cell writes `bench/results/<ts>.h<HOSTS>.{lan|wan<MS>}.*` so the
result files don't collide and the scenario is encoded in the
filename.

Override the matrix axes:

- `HOSTS_MATRIX="1 3"` — drop the 10-host cell for a faster sweep
- `WAN_MATRIX="0 50 100 200"` — sweep more WAN values

### Tear down

```bash
BENCH_HOSTS=10 scampi apply bench/teardown.scampi    # match the max provisioned
rm -rf bench/.ssh bench/ansible/venv bench/ansible/inventory.ini
```

Removes the LXCs (snapshots go with them) and cleans up the
ephemeral local state. Numbers are a snapshot in time; the durable
artifact is the reproducer in this directory.

## Why on PVE and not cloud VMs

Both tools see the same hardware and network conditions on a PVE
homelab cluster — the *delta* between scampi and ansible is what's
reported, and that delta doesn't care whether the underlying box is
DigitalOcean or a bare-metal Proxmox node. The PVE-hosted run is one
data point; anyone wanting to verify on different infra spins up
their own 3 hosts (cloud, homelab, whatever) and reruns the same
configs. The methodology page discloses the specific hardware and
network used for the published numbers.
